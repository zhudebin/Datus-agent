## 四、手工 E2E 剧本（发版前必过一次）

### E2E-1：CLI 默认链路
1. `uv sync` 升级到 0.2.7 候选。
2. 在一个干净目录 `cd ~/tmp/datus-demo` 初始化：`datus init`。
3. `datus chat`，提问 `select top 5 from lineitem limit 1`，观察 SQL 生成、执行、Ctrl+O 切换思考区。
4. 退出，再 `.sessions` + `.resume`：只看到 chat agent 的 session（#594）。
5. `.rewind` 到上一步，继续对话成功。

### E2E-2：Print Mode + Proxy
1. `datus chat --print --proxy_tools query_sql < input.jsonl > out.jsonl`
2. 外部脚本监听 stdout 的 `call-tool` 事件，将 mock 结果写回 stdin。
3. 验证 `out.jsonl` 含完整 Action / Message 序列；`--resume <id>` 可恢复。

### E2E-3：独立 API
1. `datus-api server --host 127.0.0.1 --port 8080 --stream --interactive`
2. 用 `curl -N` 对 `POST /api/v1/chat` 发流式请求。
3. `POST /api/v1/tools/search_metrics` body=`{"keyword": "..."}` 验证新 dispatch 路径。
4. `GET /api/v1/chat/sessions?subagent_id=foo` 验证过滤。

### E2E-4：Gateway 网关（飞书）
1. `datus-gateway --daemon --action start --pid-file ~/.datus/run/gateway.pid --daemon-log-file ~/.datus/logs/gateway.log`
2. `--action status` / `--action restart` / `--action stop` 全链路验证。
3. 在飞书单聊发起提问，确认增量下行文本正确（无“Hiagain!”类粘连），群聊 @bot 正常，未参与的 thread 回复被忽略。

### E2E-5：Filesystem Strict
1. 使用 `--filesystem-strict` 启动 CLI；让模型尝试 `read_file("/etc/hosts")` → 硬拒绝。
2. 切换到 `--no-filesystem-strict`，同样调用 → 出现权限 prompt；选择 allow once，后续相同路径再次 prompt。
3. Gateway / API bootstrap 读取 `agent.filesystem.strict=true`，无交互环境下外部路径请求直接拒绝，不挂起。

### E2E-6：KB 迁移验证（从 0.2.6 上来的用户）
1. 准备一份 0.2.6 的 KB：`subject/semantic_models/duckdb_local/foo.yaml`。
2. 升级到 0.2.7 直接 `datus chat` → 生成新 KB 时路径应为 `subject/semantic_models/foo.yaml`（不含 `duckdb_local/`）。
3. 跑 Release Note 提供的迁移脚本 → 旧内容归位；`@subject` 命令可识别。

### E2E-7：Auto Memory
1. `datus chat`，输入 `Remember that my project uses PostgreSQL`；观察 `.datus/memory/chat/MEMORY.md` 被写入。
2. `exit` 再启 → `MEMORY.md` 内容出现在系统提示；空态场景（手动清空文件）提示为 `empty -- no memories saved yet`。

## 六、回滚预案

1. **无法紧急修复时**：在 PyPI 标记 `0.2.7` 为 yanked，保留 `0.2.6` 作为推荐版本。
