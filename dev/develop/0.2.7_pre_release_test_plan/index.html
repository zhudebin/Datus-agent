    Datus-Agent 0.2.7 发版前测试计划（研发内部）¶

    目标：在 fstool-fix 合入 main 并打 v0.2.7 前，覆盖 Leo / Felix.Liu 自 v0.2.6 后 39 个提交的回归测试。 本文档面向研发与 QA；所有测试用例须可重复执行，优先用已有 pytest 标记体系（CI / nightly / regression）。

    一、测试范围与优先级¶

    P0（阻塞发版）¶

    模块 对应 PR 验证重点
    文件系统工具 / PathZone #578 / #588 / #561 / #587 跨 zone 读写、strict 模式拒绝、KB 路径锚定
    Auto Memory（MEMORY.md） #498 / #595 注入、空态占位、L1/L2 加载、写入权限
    REST API / Chat API Streaming #520 / #554 / #555 独立部署、流式事件顺序、interactive 透传

    filesystem_strict 三态 d6c610bf YAML / CLI 三态覆盖规则、strict 无人值守

    P1（发版前必过）¶

    二、测试分层¶

    层级 用例前缀 运行命令
    CI（零外部依赖） tests/unit_tests/ uv run pytest tests/unit_tests/ -q
    Nightly（真实 LLM） @pytest.mark.nightly uv run pytest -m nightly tests/
    Regression（DB / 多 Provider） @pytest.mark.regression uv run pytest -m "nightly or regression" tests/
    手工 E2E 下方 §4 剧本 人工

    覆盖率门槛：--cov=datus --cov-fail-under=80；diff coverage uv run diff-cover coverage.xml --compare-branch=upstream/main --fail-under=80。

    三、自动化测试用例设计¶

    下列编号对应 tests/unit_tests/... 中的新增或需补齐用例；每个 case 至少 1 条正向 + 1 条反向。

    3.1 FilesystemFuncTool / PathZone（#588 / #561）¶

    tests/unit_tests/tools/func_tool/test_filesystem_func_tool.py

    ID 场景 输入 断言
    FS-01 INTERNAL 读 read_file("./subject/semantic_models/foo.yaml") 成功；无 prompt
    FS-02 WHITELIST 写 skills write_file(".datus/skills/x.md", ...) 成功；无 prompt
    FS-03 WHITELIST memory write_file(".datus/memory/chat/MEMORY.md", ...) 成功
    FS-04 HIDDEN 读拒绝 read_file(".datus/sessions/abc.db") PathZone=HIDDEN，非 strict 下走 prompt；strict 下 PermissionDeniedException
    FS-05 EXTERNAL + strict strict=True, 读 /etc/hosts 硬拒绝，不触达 broker
    FS-06 EXTERNAL + interactive strict=False, 同上 触发权限 prompt
    FS-07 glob 递归 glob("**/*.sql") 返回按 mtime 排序；不命中 .datus/ HIDDEN
    FS-08 grep 命中 grep("SELECT", glob="*.sql") 返回含行号的 content
    FS-09 edit_file 唯一性 old_string 重复 抛错提示非唯一
    FS-10 read_file offset/limit 10000 行文件，offset=100, limit=50 返回 50 行，首行号正确
    FS-11 旧工具已移除 调用 list_directory AttributeError（保证 #561 清理彻底）

    四、手工 E2E 剧本（发版前必过一次）¶

    E2E-1：CLI 默认链路¶

    1. uv sync 升级到 0.2.7 候选。
    2. 在一个干净目录 cd ~/tmp/datus-demo 初始化：datus init。
    3. datus chat，提问 select top 5 from lineitem limit 1，观察 SQL 生成、执行、Ctrl+O 切换思考区。
    4. 退出，再 .sessions + .resume：只看到 chat agent 的 session（#594）。
    5. .rewind 到上一步，继续对话成功。

    E2E-2：Print Mode + Proxy¶

    1. datus chat --print --proxy_tools query_sql < input.jsonl > out.jsonl
    2. 外部脚本监听 stdout 的 call-tool 事件，将 mock 结果写回 stdin。
    3. 验证 out.jsonl 含完整 Action / Message 序列；--resume <id> 可恢复。

    E2E-3：独立 API¶

    1. datus-api server --host 127.0.0.1 --port 8080 --stream --interactive
    2. 用 curl -N 对 POST /api/v1/chat 发流式请求。
    3. POST /api/v1/tools/search_metrics body={"keyword": "..."} 验证新 dispatch 路径。
    4. GET /api/v1/chat/sessions?subagent_id=foo 验证过滤。

    E2E-4：Gateway 网关（飞书）¶

    1. datus-gateway --daemon --action start --pid-file ~/.datus/run/gateway.pid --daemon-log-file ~/.datus/logs/gateway.log
    2. --action status / --action restart / --action stop 全链路验证。
    3. 在飞书单聊发起提问，确认增量下行文本正确（无“Hiagain!”类粘连），群聊 @bot 正常，未参与的 thread 回复被忽略。

    E2E-5：Filesystem Strict¶

    1. 使用 --filesystem-strict 启动 CLI；让模型尝试 read_file("/etc/hosts") → 硬拒绝。
    2. 切换到 --no-filesystem-strict，同样调用 → 出现权限 prompt；选择 allow once，后续相同路径再次 prompt。
    3. Gateway / API bootstrap 读取 agent.filesystem.strict=true，无交互环境下外部路径请求直接拒绝，不挂起。

    E2E-6：KB 迁移验证（从 0.2.6 上来的用户）¶

    1. 准备一份 0.2.6 的 KB：subject/semantic_models/duckdb_local/foo.yaml。
    2. 升级到 0.2.7 直接 datus chat → 生成新 KB 时路径应为 subject/semantic_models/foo.yaml（不含 duckdb_local/）。
    3. 跑 Release Note 提供的迁移脚本 → 旧内容归位；@subject 命令可识别。

    E2E-7：Auto Memory¶

    1. datus chat，输入 Remember that my project uses PostgreSQL；观察 .datus/memory/chat/MEMORY.md 被写入。
    2. exit 再启 → MEMORY.md 内容出现在系统提示；空态场景（手动清空文件）提示为 empty -- no memories saved yet。

    六、回滚预案¶

    1. 无法紧急修复时：在 PyPI 标记 0.2.7 为 yanked，保留 0.2.6 作为推荐版本。
    2. KB 目录锚定引发用户数据误读：在 0.2.7.1 中新增一次性 datus migrate-kb 子命令，按 Release Note §5.3 的逻辑原子搬迁。
    3. Gateway daemon 存活问题：提供 --no-daemon 前台兜底，文档明示。
